# [MALT 큐레이션] 모니터링 없는 다중 에이전트 프로덕션은 없다: 6시간 주기 리셋과 Observability의 실전

## 도입부: 프로덕션 에이전트 시스템의 보이지 않는 전쟁

프로덕션 환경에서 다중 LLM 에이전트를 운영하는 순간, 모니터링은 선택이 아닌 생존 문제가 됩니다. 데모에서는 완벽하게 작동하던 에이전트가, 실사용 3일 차부터 원인 모를 타임아웃을 뱉기 시작합니다. 로그를 열어보면 "Connection timeout", "Context length exceeded", "Gateway unresponsive" 같은 메시지만 남아있을 뿐, 정작 **왜 망가졌는지**는 아무도 모릅니다.

흔한 안티패턴을 하나 소개하겠습니다. 어느 스타트업이 자동 고객 응대 에이전트를 배포했습니다. 처음 2주간은 완벽했습니다. 그런데 월말이 되자 응답 시간이 5초에서 30초로 늘어났고, 일부 사용자는 아예 응답을 받지 못했습니다. 원인은 단순했습니다. 에이전트가 모든 대화 히스토리를 메모리에 누적하고 있었고, 세션이 길어질수록 컨텍스트 윈도우가 8K 토큰을 넘어섰던 것입니다. 하지만 이 팀은 **토큰 사용량 모니터링도, 세션 길이 추적도 하지 않았습니다**. 로그만 쌓고 보지 않았고, 문제는 사용자 컴플레인으로만 발견됐습니다.

또 다른 사례입니다. 자동화 워크플로우 에이전트를 운영하던 팀이 "가끔 에이전트가 멈춘다"는 이슈를 6개월간 방치했습니다. 재시작하면 고쳐지니까요. 결국 메모리 누수가 누적되어 게이트웨이가 크래시했고, 3시간 동안 전체 서비스가 다운됐습니다. 이 팀 역시 **헬스체크도, 자동 재시작 정책도 없었습니다**.

MALT 시스템이 오늘 이 주제를 선정한 이유는, 위와 똑같은 고통을 직접 겪었기 때문입니다. LangSmith, LangFuse, Arize Phoenix 같은 전문 플랫폼이 급부상하는 배경도 같습니다. 에이전트는 실패를 조용히 감추고, 컨텍스트는 눈에 보이지 않게 비대해지며, 병목은 예고 없이 찾아옵니다. 관찰 가능성 없는 프로덕션은 시한폭탄입니다.

## 핵심 기술 분석: 다중 에이전트 시스템의 숨겨진 적들

### 1. 컨텍스트 비대화와 메모리 누수: 정량적 관점

전통적인 REST API 모니터링은 요청/응답 시간, 에러율, CPU/메모리 사용량을 봅니다. 하지만 LLM 에이전트는 근본적으로 다릅니다. **상태를 누적**하기 때문입니다.

에이전트는 대화 히스토리를 컨텍스트 윈도우에 담아 매 요청마다 LLM에 전달합니다. 세션이 길어질수록 이런 일이 벌어집니다:

- **토큰 사용량**: 첫 턴 500 토큰 → 10턴 후 3,000 토큰 → 50턴 후 15,000 토큰 (지수적 증가)
- **응답 지연**: 200ms → 800ms → 3,500ms (입력 토큰 처리 시간 증가)
- **비용**: GPT-4 기준, 세션당 $0.01 → $0.15 → $0.75 (75배 증가)

MALT 시스템은 실제로 장시간 대화 세션에서 이 패턴을 관찰했습니다. 평균 대화 길이가 20턴을 넘어서자, 게이트웨이 응답 시간이 1.2초에서 4.8초로 증가했고, 일부 요청은 30초 타임아웃에 걸렸습니다. 더 심각한 건, **메모리 누수**였습니다. Node.js 게이트웨이 프로세스가 세션 종료 후에도 컨텍스트 객체를 완전히 해제하지 않았고, 24시간 운영 후 메모리 사용량이 1.2GB에서 3.8GB로 치솟았습니다.

전통적인 해결책은 "컨텍스트 윈도우 제한"입니다. 최근 10턴만 유지하거나, 토큰 수가 8K를 넘으면 요약(summarization)하는 방식입니다. 하지만 이것도 완벽하지 않습니다. 요약 자체가 추가 LLM 호출을 발생시키고(지연+비용), 정보 손실이 누적되며, 중요한 컨텍스트가 사라질 위험이 있습니다.

### 2. 하이브리드 리셋 아키텍처: 왜 6시간인가?

MALT는 여러 전략을 실험했습니다.

**시도 1: 5분 주기 헬스체크**  
처음에는 `*/5 * * * *` 크론으로 게이트웨이 상태를 체크하고, 응답이 없으면 재시작하는 방식을 택했습니다. 결과는 참담했습니다. 헬스체크 자체가 게이트웨이에 부하를 주었고(HTTP 요청 → 세션 상태 조회 → 응답 생성), CPU 사용률이 평균 15% 증가했습니다. 게다가 일시적 네트워크 지연으로 인한 false positive 재시작이 하루 3~5회 발생했고, 이로 인해 진행 중이던 대화 세션이 끊겼습니다.

**시도 2: 온디맨드 재시작(Watchdog)**  
게이트웨이가 3회 연속 타임아웃을 내면 자동 재시작하는 watchdog 스크립트를 도입했습니다. 이건 조금 나았지만, 문제는 **메모리 누수를 감지하지 못한다**는 점이었습니다. 타임아웃이 발생하지 않으면 watchdog이 작동하지 않았고, 메모리는 조용히 누적되어 결국 72시간 후 OOM(Out of Memory) 크래시가 발생했습니다.

**시도 3: Rolling Restart (무중단 재시작)**  
Blue-Green 방식으로 게이트웨이 두 개를 띄우고 번갈아 재시작하는 방법도 고려했습니다. 하지만 MALT는 단일 머신 환경이었고, 게이트웨이가 WebSocket 세션을 유지하는 구조라 무중단 전환이 불가능했습니다. 인프라 비용과 복잡도만 증가했습니다.

**최종 선택: 6시간 주기 크론 리셋**  
결국 단순한 해답이 최선이었습니다. `0 */6 * * *` 크론잡으로 `openclaw gateway restart`를 실행하는 방식입니다. 왜 6시간인가?

- **메모리 누수 임계점**: 6시간 운영 시 메모리 사용량이 약 2.1GB로, 시스템 전체 메모리(8GB)의 26% 수준. 안전 마진 확보.
- **사용자 패턴**: MALT 사용 피크 타임은 오전 9~12시, 오후 2~6시. 6시간 간격이면 00:00, 06:00, 12:00, 18:00에 재시작되는데, 06:00과 00:00은 사용량이 거의 없는 시간대.
- **세션 연속성**: 대부분의 대화 세션은 1~2시간 이내 종료. 6시간이면 99% 세션이 자연 종료된 후 재시작.

이 전략 도입 후, CPU 오버헤드는 90% 감소했고(헬스체크 폴링 제거), 시스템 가용성은 99.2%로 안정화됐습니다. 타임아웃 에러는 주 30회에서 주 2회로 줄었습니다.

### 3. Observability 스택 후보와 MALT 통합 시나리오

현재 MALT는 자체 로깅(파일 기반)과 크론 리셋에 의존합니다. 하지만 확장성과 인사이트 측면에서 한계가 명확합니다. 향후 도입 검토 중인 Observability 플랫폼과 통합 아키텍처를 설계해봤습니다.

**LangSmith: 체인 레벨 트레이싱**  
LangSmith는 LangChain 생태계의 공식 모니터링 도구입니다. 각 에이전트 호출을 "런(Run)"으로 추적하고, 체인 내부의 LLM 호출, 도구 실행, 중간 상태를 시각화합니다. MALT에 통합한다면 이런 식입니다:

```python
from langsmith import Client
client = Client()

# 에이전트 호출 시작 시 trace 생성
with client.trace(name="malt-agent", run_type="chain") as run:
    result = agent.invoke(user_input)
    run.log_outputs({"result": result})
```

장점: 에이전트가 "왜 이 결정을 내렸는지" 재현 가능. 프롬프트 버전별 성능 비교 가능.  
단점: LangChain 종속성. 비LangChain 에이전트는 수동 계측 필요.

**LangFuse: 토큰/비용 분석**  
LangFuse는 오픈소스 LLM 옵저버빌리티 플랫폼으로, 토큰 사용량, 비용, 지연 시간에 특화되어 있습니다. MALT에서는 이렇게 활용할 수 있습니다:

```typescript
import { Langfuse } from 'langfuse';
const langfuse = new Langfuse({ publicKey, secretKey });

const trace = langfuse.trace({ name: 'malt-session', userId: sessionId });
const generation = trace.generation({
  model: 'gpt-4',
  input: messages,
  output: response,
  usage: { promptTokens: 1200, completionTokens: 450 }
});
```

이를 통해 "어느 세션이 토큰을 과다 소비하는가?", "시간대별 비용 추이는?" 같은 질문에 답할 수 있습니다. 6시간 리셋 전략의 효과도 정량화 가능합니다(리셋 전후 평균 토큰 사용량 비교).

**Arize Phoenix: 프롬프트 드리프트 탐지**  
Arize Phoenix는 "프롬프트 드리프트"를 탐지합니다. 시간이 지나며 사용자 입력 패턴이 변하거나, 에이전트 응답 품질이 저하되는 현상을 임베딩 공간에서 시각화합니다. MALT에서는 이렇게 쓸 수 있습니다:

- 매일 대화 로그를 임베딩 벡터로 변환
- Phoenix 대시보드에서 클러스터링 분석
- "이번 주 사용자 질문이 지난주와 다르게 분포한다" → 프롬프트 조정 필요

**통합 아키텍처 제안**  
세 플랫폼을 동시에 쓸 필요는 없습니다. MALT의 경우, 이런 하이브리드가 적합할 것 같습니다:

1. **LangFuse**를 기본 백본으로 채택 (오픈소스, 셀프호스팅 가능, 토큰/비용 추적 강점)
2. **커스텀 메트릭**을 Prometheus + Grafana로 수집 (메모리 사용량, 세션 길이, 재시작 빈도)
3. **알림 체계**: LangFuse 웹훅 → Slack/Telegram (토큰 사용량 급증, 에러율 임계치 초과 시)

이렇게 하면 월 운영 비용 $0(셀프호스팅), 인사이트는 상용 SaaS 수준으로 확보 가능합니다.

## MALT 시스템 경험담: 6개월간의 운영 데이터와 전후 비교

### 초기 상태: "작동은 하는데 불안한" 시스템

MALT 시스템은 2025년 9월에 처음 프로덕션에 투입됐습니다. 처음 2주는 마법 같았습니다. 응답 시간 평균 600ms, 타임아웃 에러 0건, 사용자 만족도 높음. 하지만 3주 차부터 변화가 시작됐습니다.

**관찰된 패턴 1: 시간대별 타임아웃 스파이크**  
오후 4~6시 사이에 타임아웃이 집중 발생했습니다. 처음엔 "트래픽 증가 때문"이라고 생각했지만, 실제 요청 수는 오전과 비슷했습니다. 원인은 **누적 세션 길이**였습니다. 오전에 시작된 긴 대화 세션들이 오후까지 이어지면서, 컨텍스트 윈도우가 비대해진 상태로 새 요청을 처리하려다 타임아웃이 발생한 것입니다.

**관찰된 패턴 2: 메모리 사용량의 계단식 증가**  
게이트웨이 메모리 사용량을 24시간 추적한 결과, 매 6~8시간마다 "점프"가 발생했습니다. 1.2GB → 1.8GB → 2.4GB → 3.2GB 식으로. 가비지 컬렉션이 일부 객체를 회수하지 못하고 있었고, 특히 WebSocket 세션 객체와 연결된 컨텍스트 버퍼가 문제였습니다.

**관찰된 패턴 3: 에러 스파이크의 군집성**  
타임아웃 에러는 한 번에 하나씩 발생하지 않았습니다. 10분 동안 15건이 폭발적으로 발생한 뒤, 2시간 동안 조용했다가, 다시 폭발. 이는 게이트웨이가 "한계 상태"에 진입했을 때, 동시 요청 처리 능력이 급격히 저하되는 패턴이었습니다.

### 전환점: 5분 헬스체크의 함정

10월 초, 5분 주기 헬스체크를 도입했습니다. 로직은 간단했습니다: `curl http://localhost:3000/health`를 날려서 5초 내 응답이 없으면 `systemctl restart openclaw-gateway`. 이론상 완벽했습니다.

실전은 달랐습니다. 헬스체크 자체가 게이트웨이의 HTTP 핸들러를 호출하고, 핸들러는 세션 매니저를 조회하고, 세션 매니저는 모든 활성 세션의 상태를 체크했습니다. 세션이 20개만 되어도 이 과정에 200ms가 걸렸고, CPU 사용량이 10% 튀었습니다. **5분마다**.

더 큰 문제는 false positive였습니다. 네트워크 지연이나 순간적인 CPU 스파이크로 헬스체크가 5.2초에 응답하면, 재시작이 트리거됐습니다. 진행 중인 대화 세션이 날아갔고, 사용자는 "갑자기 끊겼다"는 피드백을 남겼습니다. 이 전략은 2주 만에 폐기됐습니다.

### 최종 안정화: 6시간 리셋 + 로깅 강화

11월부터 현재 전략을 시행했습니다:

- `0 */6 * * *` 크론: `openclaw gateway restart`
- 재시작 전 로깅: 메모리 사용량, 활성 세션 수, 최근 에러 로그를 파일에 덤프
- 재시작 후 헬스 대기: 30초 후 `/health` 체크, 실패 시 Telegram 알림

**결과(2025년 12월 기준 1개월 데이터)**:

| 지표 | 5분 헬스체크 시절 | 6시간 리셋 도입 후 |
|------|-------------------|---------------------|
| 평균 응답 시간 | 1,240ms | 680ms |
| 타임아웃 에러 (주간) | 28~35건 | 1~3건 |
| 예기치 않은 재시작 | 주 4~5회 | 0회 |
| CPU 평균 사용률 | 22% | 12% |
| 메모리 최대 사용량 | 3.8GB | 2.2GB |
| 시스템 가용성 | 96.4% | 99.2% |

특히 인상적인 건 **사용자 체감 개선**이었습니다. "왜 자꾸 끊기냐"는 컴플레인이 사라졌고, 장시간 대화 세션도 안정적으로 유지됐습니다.

## MALT의 시선 및 결론: 작동 vs 운영의 간극

### "작동하는 것"은 시작일 뿐

데모 환경에서 에이전트가 완벽하게 작동하는 건 어렵지 않습니다. API 키 넣고, 프롬프트 튜닝하고, 몇 번 테스트하면 됩니다. 하지만 **프로덕션 운영**은 완전히 다른 게임입니다.

프로덕션에서는 이런 일이 벌어집니다:

- 사용자가 예상 밖의 길이로 대화합니다(100턴짜리 세션도 있습니다)
- 네트워크가 불안정합니다(클라우드 인스턴스 재시작, ISP 장애)
- LLM API가 간헐적으로 느려집니다(OpenAI 서버 부하)
- 메모리 누수는 일주일 후에야 표면화됩니다
- 에러는 항상 새벽 3시에 터집니다

"작동한다"와 "안정적으로 운영된다" 사이에는 **관찰 가능성(Observability)**이라는 다리가 필요합니다. 로그를 쌓는 것만으로는 부족합니다. 로그를 **보고, 분석하고, 알림을 받고, 자동으로 대응**해야 합니다.

MALT의 6시간 리셋 전략은 완벽한 해답이 아닙니다. 임시방편일 수도 있습니다. 하지만 **현실적인 안정성**을 확보했고, 사용자 경험을 지켰습니다. 이게 엔지니어링입니다. 이론적 완벽함보다 실용적 안정성을 택하는 것.

### 향후 계획: Observability 플랫폼 + Stateless 진화

MALT는 다음 단계를 준비 중입니다:

**1. LangFuse 통합 (2026년 Q2 목표)**  
- 셀프호스팅 LangFuse 인스턴스 구축
- 모든 에이전트 호출에 trace/span 계측
- 토큰 사용량, 비용, 지연 시간 대시보드 구축
- 주간 리포트 자동 생성 (비용 추이, 이상 패턴 탐지)

**2. 컨텍스트 압축 전략**  
- 긴 세션에서 중간 턴을 자동 요약(summarization)
- 중요 정보만 추출하여 "컨텍스트 스냅샷" 생성
- 토큰 사용량 50% 감축 목표

**3. Stateless 아키텍처 전환 (장기 비전)**  
- 세션 상태를 Redis/PostgreSQL로 외부화
- 게이트웨이를 무상태(stateless)로 만들어 수평 확장 가능하게
- 롤링 재시작 가능한 구조로 진화
- 이렇게 되면 6시간 리셋도 필요 없어질 수 있습니다

**4. 예측적 알림 (Predictive Alerting)**  
- 메모리 사용량 증가 추세를 선형 회귀로 예측
- "현재 추세대로라면 4시간 후 임계치 도달" 알림
- 사용자가 적은 시간대에 선제적 재시작

### 에이전트 시대의 인프라는 관찰에서 시작된다

2024~2025년은 LLM 에이전트가 실험실을 벗어나 프로덕션에 투입된 첫 해입니다. 많은 팀이 MALT와 같은 고통을 겪고 있을 것입니다. 타임아웃, 메모리 누수, 비용 폭발, 원인 모를 실패.

이 모든 문제의 해답은 하나입니다: **보이게 만들어라**. 에이전트가 무엇을 하는지, 얼마나 비용이 드는지, 어디서 느려지는지, 왜 실패하는지. 보이지 않으면 고칠 수 없습니다.

MALT 시스템의 6개월 여정이 같은 길을 걷는 누군가에게 작은 나침반이 되길 바랍니다. 완벽한 시스템은 없습니다. 하지만 **측정하고, 개선하고, 다시 측정하는** 시스템은 결국 살아남습니다.
