# PHM Agentic 검증 파이프라인 실패 패턴 7가지와 복구 전략

PHM 운영에서 모델 정확도보다 더 치명적인 것은 **검증 파이프라인 불안정**입니다. 특히 Agentic 워크플로를 도입하면 자동화는 빨라지지만, 데이터/스키마/환경 불일치가 연쇄적으로 확산되기 쉽습니다. 이번 글은 현장에서 자주 발생하는 실패 패턴과 복구 전략을 압축 정리합니다.

## 실패 패턴 1) 데이터 타입 불일치
- 증상: 시계열 모델에 이미지/아티팩트 데이터셋이 들어감
- 복구: 알고리즘별 `expected_data_type` 강제 검증

## 실패 패턴 2) 수동 지표 입력
- 증상: ROC/F1를 사람이 직접 입력해 재현성 붕괴
- 복구: 결과 파일(JSON/DB) 기반 자동 반영만 허용

## 실패 패턴 3) 모델-검증 연결 분리
- 증상: KPI는 보이지만 어떤 모델/버전 결과인지 모름
- 복구: 레지스트리에 `model_id`, `version`, `evaluated_at` 필수화

## 실패 패턴 4) 학습/검증 데이터셋 혼합
- 증상: train/test 구분이 보고서에서 사라짐
- 복구: `train_dataset` / `validation_dataset` 별도 저장

## 실패 패턴 5) 실패 원인 비가시화
- 증상: "검증 실패"만 보이고 원인이 없음
- 복구: 실패 메시지 + source file path + 누락 필드 표시

## 실패 패턴 6) 결과 스키마 드리프트
- 증상: 어떤 실행은 AP가 없고 어떤 실행은 confusion이 없음
- 복구: 필수 키(ROC-AUC, F1, AP, n_samples, confusion) 스키마 검증

## 실패 패턴 7) 운영 환경 차이
- 증상: 로컬에선 통과, 운영에선 실패
- 복구: 환경 점검(preflight) 자동화 + 의존성 고정

---

## 실무 적용 체크리스트
1. 데이터 타입 가드 적용
2. 실측 자동 반영
3. 모델/버전/시점 추적
4. train/validation 분리
5. 실패 로그 표준화
6. 결과 스키마 검증
7. preflight 루틴 고정

---

## 결론
Agentic PHM의 핵심은 "자동화 속도"가 아니라 **검증 신뢰성**입니다. 자동화는 실패를 없애주지 않습니다. 대신 실패를 빠르게 감지하고, 재현 가능하게 복구하는 구조를 만들 때 비로소 운영 품질이 올라갑니다.
