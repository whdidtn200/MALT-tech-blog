# PHM Mission Control 실전 설계: 모델-검증 연계, 데이터 타입 거버넌스, 자동 실측 반영

산업 현장에서 PHM(Prognostics and Health Management) 대시보드는 흔히 “보기 좋은 화면”으로 끝납니다. 하지만 운영 중 가장 중요한 질문은 UI가 아니라 다음 세 가지입니다.

1. **이 수치가 어떤 모델의 결과인가?**
2. **학습 데이터셋과 검증 데이터셋이 정확히 분리되어 추적되는가?**
3. **수동 입력 없이 실측 결과가 자동 반영되는가?**

이번 글에서는 위 세 가지를 충족하도록 PHM Mission Control을 재설계한 과정을 공유합니다. 핵심은 **모델-검증 강결합 구조**, **데이터 타입 기반 검증 가드**, **실측 파일 단일 진실원(single source of truth)** 입니다.

---

## 1) 문제 정의: "검증"이 아니라 "표시"만 하던 대시보드

초기 대시보드에서 자주 발생하는 문제는 다음과 같습니다.

- 검증 KPI가 전역 1세트로 표시되어 모델별 추적이 불가능
- 수동 입력 필드(ROC/F1)에 의해 임의값 유입 가능
- 학습/검증 데이터셋이 혼재되어 감사 추적성 부족
- 이미지 기반 데이터셋을 시계열 파이프라인으로 잘못 검증 시도

결론적으로, 대시보드는 화려하지만 **거버넌스 관점에서는 신뢰 불가 상태**였습니다.

---

## 2) 목표 아키텍처

### 2.1 모델별 검증 이력 구조

알고리즘 레지스트리를 아래처럼 재구성했습니다.

```json
{
  "id": "lstm-autoencoder-001",
  "name": "LSTM Autoencoder Diagnostics",
  "validations": [
    {
      "train_dataset": "IEEE PHM 2012 Test Set",
      "dataset": "Uniwear PHM Dataset Bundle",
      "tool": "scikit-learn",
      "evaluated_at": "2026-03-05T17:03:24",
      "metrics": {
        "roc_auc": 0.8325,
        "best_f1": 0.5660,
        "average_precision": 0.6059,
        "threshold": 0.8783,
        "n_samples": 7420
      },
      "confusion": { "TP": 1001, "FP": 1027, "FN": 509, "TN": 4883 }
    }
  ]
}
```

이 구조의 장점은 명확합니다.

- 모델 단위 버전/검증이력 분리
- 학습 데이터셋(`train_dataset`)과 검증 데이터셋(`dataset`) 동시 추적
- 검증 재현성(파일/시간/툴) 확보

---

### 2.2 데이터 타입 기반 검증 가드

데이터셋마다 `data_type` 메타를 부여했습니다.

- `timeseries`
- `image`
- `tabular`
- `artifact`

그리고 알고리즘별 기대 타입을 매핑합니다.

- `lstm-autoencoder-001` → `timeseries`
- `vae-anomaly-001` → `timeseries`

검증 시 타입이 맞지 않으면 저장 단계에서 즉시 차단합니다.

```text
검증 실패: 데이터 형태 불일치 (알고리즘=timeseries, 선택 데이터셋=image)
```

이 한 줄 가드만으로도 운영 사고(잘못된 데이터셋 검증 저장)를 크게 줄일 수 있습니다.

---

### 2.3 자동 실측 반영 (수동 입력 제거)

가장 중요한 변경은 **ROC/F1 수동 입력 제거**입니다.

검증 결과는 항상 최신 실측 파일에서 로드합니다.

- LSTM: `lstm_ae_threshold_summary.json`
- VAE: `vae_threshold_summary.json`

그리고 저장 시 아래 지표를 모두 자동 반영합니다.

- ROC-AUC
- Best F1
- Average Precision
- Threshold
- N samples
- Confusion Matrix

즉, “대시보드 입력값”이 아니라 “실측 산출물”이 진실원이 됩니다.

---

## 3) 운영 관점에서의 개선 포인트

### 3.1 성공/실패 상태의 명시

검증 버튼 클릭 후 반드시 다음을 표시하도록 설계했습니다.

- 성공/실패 상태
- 실패 원인(파일 없음/매핑 없음/지표 누락/타입 불일치)
- 참조한 결과 파일 경로(`source_file`)

운영자는 더 이상 "왜 안 됐는지"를 로그에서 역추적할 필요가 없습니다.

---

### 3.2 지표 표시 계층 분리

대시보드 상단 KPI와 레지스트리/리더보드를 분리했습니다.

- **KPI 카드**: 선택 알고리즘 최신 검증값
- **검증 추이 바**: 모델별 ROC/F1/AP 상대 비교
- **리더보드**: 데이터셋+AP 기반 성능 순위
- **레지스트리**: 학습/검증 데이터셋 + 최근 검증 시점

이렇게 분리하면 운영자가 “현재 상태”와 “비교 관점”을 동시에 확인할 수 있습니다.

---

## 4) 실제 지표 반영 예시

LSTM Autoencoder 실측 결과는 다음과 같이 반영되었습니다.

- ROC-AUC: **0.8326**
- Best F1: **0.5660**
- Average Precision: **0.6059**
- Threshold: **0.8783**
- Samples: **7420**
- Confusion: TP 1001 / FP 1027 / FN 509 / TN 4883

중요한 점은 “수치 자체”보다 **수치가 저장되는 방식**입니다. 
모든 지표가 모델/데이터셋/검증시점과 함께 남아야, 나중에 성능 저하 원인을 정확히 추적할 수 있습니다.

---

## 5) 데이터셋 "건수"의 의미도 명확히 해야 한다

운영 중 자주 발생하는 오해가 하나 있습니다.

- 화면의 `건수`는 샘플 수가 아니라 파일 수인가?

이를 명확히 분리하지 않으면 의사결정 오류가 발생합니다. 예를 들어,
- 이미지 기반 결함셋은 파일 2개일 수 있지만
- 시계열 CSV 한 파일 안에는 수천~수만 샘플이 존재할 수 있습니다.

따라서 차기 버전에서는 아래 분리를 권장합니다.

- 파일 건수
- 샘플 건수(행/윈도우/이미지 수)

---

## 6) 결론: PHM 대시보드의 본질은 "거버넌스"

PHM Mission Control에서 진짜 중요한 것은 UI가 아닙니다.

- 모델별 검증의 추적성
- 데이터 타입 일치성
- 실측값 자동 반영
- 실패 원인 가시성

이 네 가지가 갖춰져야 대시보드는 “시각화 도구”를 넘어 “운영 통제면(Control Plane)”이 됩니다.

현장 시스템은 완벽한 모델 하나보다, **재현 가능하고 감사 가능한 검증 체계**가 먼저입니다.
